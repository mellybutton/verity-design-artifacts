Design Principles

How we design
at Verity

Five convictions that guide every decision — layout, language, interaction, and architecture.

01
Surface the signal, hide the system

Verity's job is to extract knowledge from busy humans. Every UI decision must reduce cognitive load, not add to it. Chrome disappears; meaning remains. We show people what they've captured, not the machinery that captured it.

02
Structured but not rigid

Rule logic is inherently conditional and contextual. Our UI reflects that: clear structure (IF/THEN/BECAUSE) without forcing unnatural formality. Language stays human-scale. Components adapt to ambiguity rather than fighting it.

03
Trust is earned incrementally

Automated capture tools can feel intrusive. We build trust through transparency: showing sources, surfacing confidence scores, making human review a first-class step. Every automated action has a visible seam. Users can always see why.

04
Calm technology, urgent output

The interface should feel unhurried and precise — like a well-run meeting, not a notification swarm. But the outputs (Jira tickets, prompts, specs) must be action-ready and time-efficient. Calm inputs, sharp outputs.

05
Make the invisible visible — once

Hidden business logic is Verity's origin story. Our design mission is to surface that logic beautifully — but not to lecture. We reveal structure the first time a user needs it, teach through interaction, then get out of the way. Onboarding is embedded, not bolted on.

06
AI suggestions are helpful, honest, and harmless

Every AI-generated suggestion in Verity — elicitation questions, adaptive follow-up probes, quality chips, inline nudges — is governed by three constraints. Helpful: it must serve the practitioner's cognition, not throughput. A suggestion that fills a BECAUSE field faster without improving reasoning quality is not helpful. Honest: every AI action accurately represents what the AI did and didn't do. AI authorship is always labeled. Confidence is surfaced, not suppressed. Rule candidates are always candidates until a human confirms them. Harmless: no suggestion pattern is allowed to erode the quality of expert reasoning over time. Every pattern must survive the question: does this become harmful at session 500, not just session 1? See verity-ai-suggestion-principles-h3-v1 for the full framework.

Do's & Don'ts

✓ Do
  • ✓Use IF/THEN/BECAUSE structure for all rule language
  • ✓Show confidence and source attribution on captured rules
  • ✓Keep actions visible — never hide primary CTAs in menus
  • ✓Use semantic color for status consistently
  • ✓Write microcopy in plain language, not system language
  • ✓Give human review a prominent stage in every workflow
  • ✓Animate sparingly — only when motion carries meaning
  • ✓Ensure 4.5:1 contrast ratio minimum on all body text
✗ Don't
  • ✗Use jargon ("elicitation", "ontology") in UI-facing copy
  • ✗Auto-send outputs to Jira without a review step
  • ✗Stack more than 3 levels of navigation depth
  • ✗Use red for anything other than errors / danger actions
  • ✗Let empty states be dead ends — always offer a next action
  • ✗Use Fraunces for body text — display only
  • ✗Animate on every interaction — reserve for milestone moments
  • ✗Use placeholder text as a label replacement

Purposeful

Each animation communicates: entering, exiting, confirming, loading. It should answer "what just happened?"

Respect prefers-reduced-motion. All animations must degrade gracefully to instant transitions.

Consistent

Same actions use the same curve and duration. Predictability builds interface fluency over time.

Resources · Incident Documentation

Safari Dark Mode

Why [data-theme="dark"] fails on iOS Safari, what happens when it does, and the hardcoded workaround pattern to use for all dark-first Verity surfaces.

⚠ Incident · Feb 2026
Verity story/journey page rendered in light mode on iOS Safari despite data-theme="dark" being set on the HTML element.
Result: dark text colors on light background, washed-out serif headings, invisible strong tags, grid pattern bleeding through hero — all text contrast failed WCAG AA. Took 3 iterations to resolve.

Root Cause

Safari on iOS (and some WebView contexts including Claude.ai's file preview) does not reliably honor CSS attribute selectors like [data-theme="dark"] as a theming mechanism. The browser applies its own color-scheme inference based on system preferences, and CSS variable values that depend on the attribute selector cascade may be overridden or ignored.

Additionally, Safari aggressively applies its own background color to html and body unless those values are set explicitly with literal color values — not CSS variables.

What breaks
/* ❌ FAILS on iOS Safari — variable not resolved from attribute selector */
[data-theme="dark"] {
  --bg-page: #0C0F18;
  --text-primary: #F0F2F6;
}

html { background: var(--bg-page); }   /* Safari ignores this */
body { color: var(--text-primary); }   /* Falls back to system default */

/* Result: light background, dark-mode text colors → catastrophic contrast failure */

The Fix — Hardcode Dark Surfaces

For any Verity surface that is designed as dark-first (not theme-switchable), bypass the CSS variable system entirely for background and text on html, body, and every top-level section. Use literal hex values.

CSS — Dark-First Pages
/* ✅ CORRECT — literal values, no variable dependency */
html {
  background: #0C0F18; /* --bg hardcoded, not via variable */
}
body {
  background: #0C0F18;
  color: #F0F2F6;      /* --text hardcoded */
}

/* Every top-level section also gets explicit background */
section {
  background: #0C0F18;
}

/* Hero and any full-bleed areas — explicit, not inherited */
.hero {
  background: #0C0F18;
}
CSS — Surface & Elevated Tokens
/* For component backgrounds that use --surface or --elevated,
   also hardcode these in the :root rather than in [data-theme] */

:root {
  /* Dark values promoted to root for dark-first pages */
  --bg: #0C0F18;
  --surface: #141824;
  --elevated: #1C2233;
  --subtle: #1E2438;
  --muted: #252C44;
  --border: rgba(255,255,255,.07);
  --border-strong: rgba(255,255,255,.13);
  --text: #F0F2F6;
  --text-2: #9AA3B5;
  --text-3: #4A5369;
}

/* DO NOT rely on [data-theme="dark"] for dark-first surfaces.
   That selector is only reliable for togglable light/dark UIs
   in trusted browser contexts (desktop Chrome, Firefox). */

Text Contrast Checklist

Beyond background, Safari's rendering of text over complex backgrounds (radial glows, grid patterns) caused additional contrast failures. Rules to follow:

✓
Always set position:relative;z-index:2 on hero text
Absolute-positioned glow and grid elements must sit behind text in stacking context. Without explicit z-index on text containers, Safari stacking order is unpredictable.
✓
Never use text-shadow to solve contrast
Text shadows were tried and failed — they compound the visual noise on light backgrounds and look broken in light mode fallbacks. Fix the background, not the text.
✓
Hardcode strong tag color in hero contexts

Inline AI Suggestions

Verity's AI doesn't just capture rules — it helps people think more clearly about what they actually know. These are the patterns for surfacing that guidance inline, without interrupting flow.

Philosophy

Suggest, never prescribe

The AI offers a reframe, not a correction. Language like "consider" and "you might mean" keeps the human as the authority on their own knowledge. The AI is a thinking partner, not an editor.

Show the reasoning

Every suggestion surfaces why it was triggered — the specific vagueness, missing element, or logical gap. Users who understand the reason engage with the suggestion; users who just see "edit this" dismiss it.

One suggestion at a time

Never stack more than one active suggestion per clause. Multiple simultaneous suggestions signal that the rule needs to be rewritten from scratch — at that point, escalate to a session rather than in-place suggestions.

H3 Governing Rules — Build Reference ✦ new in v2.5

Build-level translation of the H3 governing framework. Every AI-facing UI decision must satisfy all three before shipping. Full framework: verity-ai-suggestion-principles-h3-v1.

Helpful

A suggestion is helpful if it moves the practitioner closer to articulating genuine reasoning — not if it makes the field faster to fill.

  • ✗Never optimize for BECAUSE submission speed alone
  • ✗Never use completion rate as the only success metric
  • ✓Quality chips measure genuine reasoning signal, not keyword count
  • ✓One follow-up probe maximum — never stack
Honest

Every AI-generated output accurately represents what the AI did and didn't do. Uncertainty is surfaced, not suppressed.

  • ✗Never suppress ai_attempted_because label
  • ✗Never present low-confidence extraction as precise
  • ✓Rule candidates always labeled as candidates until confirmed
  • ✓AI BECAUSE drafts always labeled + framed as critique targets
Harmless

No suggestion pattern is allowed to erode expert reasoning quality over time. Ask: does this become harmful at session 500?

  • ✗Never show example BECAUSE fields — they produce mimicry
  • ✗Scaffold prompts hidden by default; never pre-filled
  • ✓Save is never blocked by any AI suggestion
  • ✓Contrastive prompts are the anti-habituation mechanism — never replace them
